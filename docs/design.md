# Sunoパ！制作管理ツール 設計図

## 目的

Sunoパ！の放送準備から記事化、SNS投稿、画像制作、音源管理までをひとつの管理画面で扱えるようにする。

このツールは、Codexへ毎回バラバラに情報を渡すのではなく、放送回ごとに必要情報を集約し、記事作成に必要な「制作パック」を作ることを目的とする。

## 作りたいもの

```text
Sunoパ！制作管理ツール
├─ 放送回管理
├─ 複数フォーム作成
├─ 回答管理
├─ ゲスト情報管理
├─ リスナー応募曲管理
├─ パーソナリティ曲管理
├─ サムネ/画像制作管理
├─ 音源ファイル管理
├─ 記事作成パック生成
├─ SNS投稿文生成
└─ 将来: ポン出しアプリ連携
```

## 基本思想

- スプレッドシートやフォーム回答を「正本」として扱う。
- べるぼ☂が手入力する項目は、原則として自分の紹介曲URLだけにする。
- ゲスト情報、ゲスト紹介曲、リスナー応募曲、NG事項、表記ルール、音源ファイルはスプレッドシートから取り込む。
- 記事に出す情報と、運営だけが見る情報を分ける。
- ゲスト回、通常回、リスナー応募回を同じ仕組みで扱えるようにする。
- Codexに渡す情報は、必要項目だけを抽出して整理する。
- 画像や音源は、放送回単位で整理する。
- 最初から全部自動化しない。まずは制作パックを作るところまでを確実にする。

## 取り込み優先設計

基本操作は「取り込み」タブから始める。

```text
1. 放送回を選ぶ
2. ゲストアンケートシートを取り込む
3. リスナー応募曲シートを取り込む
4. パーソナリティ曲シートを取り込む
5. べるぼ☂の曲URLだけ手入力する
6. 不足項目だけ確認する
7. Codex記事作成パックを生成する
```

現時点では、公開Google Sheets CSV/export URLまたはCSVファイルを取り込む。
共有フォームは、SNSやゲスト向けには `#/r/{公開ID}` の短いURLを使う。短いURLを有効化するためのCodex依頼文をツールからコピーできる。バックアップ用途では、圧縮した `#/s/...` 形式の長いURLも使える。WAV/MP3アップロードは、回答JSONに音源データを同梱して運営側が読み込み、回答画面からダウンロードする。
回答保存Webhook URLを設定した場合は、共有フォームからGoogle Apps Scriptなどの受信口へ回答JSONをPOSTできる。Drive保存の実体はApps Script側で行う。
privateなGoogle Sheets OAuth、Drive API直結、ログイン付き同期は次フェーズ。

## 質問項目の設計

質問ごとに属性を持たせる。

```text
question_id
質問文
回答形式: 短文 / 長文 / URL / 楽曲 / X連絡ブロック / 選択式 / チェックボックス / ファイル単体
必須
公開してOKなプロフィール
記事で紹介してほしい内容
記事/SNSで触れないこと・表記ルール
制作側だけに共有するメモ
SNS投稿に使ってOK
漫画/画像案に使ってOK
表示順
```

この分類が重要。

```text
公開してOKなプロフィール: 記事やSNSの紹介文に使ってよい基本情報
記事で紹介してほしい内容: 本文の中心素材にするエピソードや楽曲の想い
記事/SNSで触れないこと・表記ルール: NG話題、表記、関係性、肩書きなどの厳守ルール
制作側だけに共有するメモ: 記事本文には載せない確認用メモ
SNS投稿に使ってOK / 漫画・画像案に使ってOK: 記事後の展開に使う素材
```

楽曲形式を選んだ質問は、回答者には次の3項目セットとして表示する。

```text
〇紹介する楽曲
・楽曲名
・楽曲URL: YouTube / Sunoのみ。その他のURLは入力時と送信時に弾く
・楽曲をWAVかMP3でアップロード
・プレビュー確認: アップロード音源、YouTube、Suno埋め込み可能URLをその場で確認
```

回答JSONを読み込むと、回答管理に保存しつつ楽曲タブにも1曲として反映する。
読み込み後の添付音源は回答管理画面でもプレビュー再生できるようにする。

X連絡ブロックを選んだ質問は、回答者には次の項目セットとして表示する。

```text
連絡用Xアカウント
・Xアカウント入力: @handle / handle / x.com URLに対応し、記事用にはURL化できる形で保存する
・連絡のためにフォローが必要であることの説明文
・べるぼ☂をフォローするボタン: https://x.com/bellbo13
・かなめ🦐をフォローするボタン: https://x.com/kaname_mbembe
・フォロー済み確認
・DM連絡OK確認
```

XアカウントとX URLは記事や紹介に使える公開情報として扱う。フォロー済み確認やDM連絡OK確認は運営確認用としてJSONに残す。

現MVPの共有フォームは、短い `#/r/{公開ID}` をメインにする。ツールからCodex用依頼文をコピーし、Codexが `public/shared-forms/{公開ID}.json` を配置すると有効になる。予備として、フォーム定義を圧縮してURLに含める `#/s/...` 方式も残す。
回答保存Webhook URLが未設定の場合、回答者は入力後に回答JSONをコピーまたはダウンロードし、運営側が回答管理画面から読み込む。
回答保存Webhook URLが設定されている場合、フォームは回答JSONを受信口へ送信する。GitHub Pages単体ではDriveへ書き込めないため、Google Apps Scriptなどの受信口がDriveフォルダー保存を担当する。

## 判断メモ

- Googleフォームではなく自前フォームを優先する。理由は、質問ごとに「記事に使う」「内部確認のみ」「NG」などの属性を持たせたいから。
- スプレッドシートは正本として有効だが、長期的にはツール内DBを正本にする方が管理しやすい。
- 画像生成は後回し。まずは素材と制作ステータスを管理する。
- 音源のZIP一括DLは主導線にしない。放送回フォルダー整理を主導線にする。
- SE_Pon連携は後段。まずは取り込みデータの仕様だけ意識しておく。
